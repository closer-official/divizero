# OS③ 案件検証OS Ver3.0
（旧：失注分析OS v2.0）

---

## Ver3.0での設計転換

旧版は「失注案件のみ」を分析対象としていた。これには2つの致命的欠陥があった。

1. **受注案件を分析しないため、勝ちパターンが蓄積されない**（失注理由は分かるのに受注理由が分からない）
2. **「提案前に消えた案件」を分類する型がなく、待ちすぎ失注が見えない**（保守バイアスは「提案して断られる」ではなく「提案に到達せず消える」形で現れる）

新版では、**クローズした全案件（受注・失注・未到達・SKIP抜き打ち）を検証対象とする。**

「動いてから判断する」体制において、このOSがシステム全体の学習エンジンである。OS①②は軽くした分、ここに最も時間をかける。

# PART2：クローズタイプ分類

## 受注タイプ（新設）

| タイプ | 内容 |
|---|---|
| W-A | 顕在課題への直行（FTが機能） |
| W-B | 関係構築からの相談化（NTが機能） |
| W-C | 相手起点（向こうから相談された） |
| W-D | その他（内容を記述） |

## 失注タイプ

| タイプ | 内容 |
|---|---|
| TypeA | タイミング不一致（相手都合・時期） |
| TypeB | 信頼不足 |
| TypeC | 営業警戒（営業認定された） |
| TypeD | ニーズ不足（そもそも困っていなかった） |
| TypeE | 温度不足（興味はあったが熱が足りなかった） |
| TypeF | 対象外（売るべき相手ではなかった＝OS①除外漏れ） |
| TypeG | 提案過早（攻めすぎ） |
| TypeH | 提案過遅（待ちすぎて温度が下がった後に提案） |
| **TypeJ（新設）** | **未到達失注：提案前にフェードアウト。どのステップで消えたかを必ず記録（S1消滅/S2消滅/S3消滅/S4消滅）** |
| **TypeK（新設）** | **停滞失注：こちらの行動遅延（返信遅れ・7日放置等）が直接原因** |
| TypeI | 不明 |

**TypeG＋Hが「攻めすぎ系」、TypeJ＋K＋Hが「待ちすぎ系」。この比率がシステム調整の根拠になる（PART5参照）。**

# PART3：個別検証（1案件ごと）

## 3-1. 事前仮説の答え合わせ（最重要）

- OS①の事前仮説は当たっていたか：的中 / 部分的中 / 外れ / 検証不能
- 実際の相手の課題・関心は何だったか
- 仮説が外れた場合、接触前のどの情報を見れば当てられたか（→ OS①の精度改善材料）

## 3-2. タイミング検証

- もっと早く動いていたら結果は変わったか：YES / NO ＋ 理由
- もっと遅く動いていたら結果は変わったか：YES / NO ＋ 理由
- ベストタイミングは実際の行動より：前 / 同じ / 後
- 赤信号の見逃しはあったか：有（引用）/ 無
- 赤信号の誤検知（実は止まる必要がなかった）はあったか：有 / 無

## 3-3. 相手視点分析（旧版から維持・簡素化）

- 相手は最初、自分を何だと思っていたか
- 最後、何だと思われたか
- 役割認識が変わった地点（具体的に引用）
- 警戒が発生した地点と理由（発生した場合のみ）
- 相手は何にお金を払う気があったか（HP / LP / 導線 / 集客 / SNS / AI活用 / 信頼形成 / 何もなし）

## 3-4. 市場データ抽出（指示書の市場監査に対応）

- この相手が本当に欲しかったものは何か
- 受注の場合：購入の決め手は何か（相手の言葉で）
- 失注の場合：何があれば買った可能性があるか
- 商材とニーズのズレ：無 / 有（内容）

## 3-5. 再現性検証（指示書の属人化監査に対応）

- この結果（受注/失注とも）は再現可能か：YES / NO
- 結果を分けた要因は：手順 / 経験 / 感覚 / 運
- 業務委託営業マンが同じ手順で同じ結果を出せるか：YES / NO ＋ 理由
- 標準手順に追加すべきルールが見つかったか：有（内容）/ 無

## 3-6. 再アプローチ判定（失注案件のみ）

- 再アプローチ可能性：S（十分可能）/ A（条件付き）/ B（困難）/ C（非推奨）/ D（不可）
- 推奨待機期間：1週間 / 1ヶ月 / 3ヶ月 / 6ヶ月以上 / 非推奨
- 再接触時の入り方（具体的に）
- 絶対にやってはいけないこと（具体的に）

## 3-7. 個別結論

- 失注理由 or 受注理由：1文
- 最大の学び：1文
- 次回同タイプへの最適行動：1文
- 学習価値：0〜100

# PART4：ルーブリック蓄積（標準化資産の構築）

検証のたびに、判断基準のアンカー（実例）を蓄積する。これが将来の業務委託教育教材になる。

記録例：

- 「前進してよい返信」の実例（実ログから引用して蓄積）
- 「赤信号R1（返信短縮）」の実例
- 「営業認定された瞬間」の実例
- 「相談化の兆候」の実例
- 「刺さったDM移行文」「滑ったDM移行文」の実例

**主観スコアを数値で定義するのではなく、実例で定義する。** 「温度4とは何か」を説明するより「温度4の返信はこれ」を見せる方が再現される。

# PART5：集計セクション（クローズ10件ごとに必ず実行）

## 5-1. ファネル集計

| 指標 | 計算 | 今回値 |
|---|---|---|
| DM移行率 | S2到達 ÷ 接触開始 | |
| 事業開示率 | S3到達 ÷ S2到達 | |
| ヒアリング率 | S4到達 ÷ S3到達 | |
| 提案率 | S5到達 ÷ S4到達 | |
| 受注率 | 受注 ÷ S5到達 | |
| 平均提案到達日数 | | |

## 5-2. 保守バイアス判定（このOSの核心）

- 攻めすぎ系失注（TypeG）：◯件
- 待ちすぎ系失注（TypeH＋J＋K）：◯件

| 比率 | 判定 | システム調整 |
|---|---|---|
| 待ちすぎ系が2倍以上 | 保守バイアス確定 | OS②の進行ラダーを1ステップ前倒し（例：リプ往復2回→1回でDM移行） |
| ほぼ同数 | バランス | 現状維持 |
| 攻めすぎ系が2倍以上 | 前進過剰 | OS②の進行ラダーを1ステップ後ろ倒し |

## 5-3. トラック検証

- FT案件の受注率 vs NT案件の受注率
- SKIP案件の抜き打ち検証：SKIPした中から2〜3件を後日観察し、「実は接触すべきだった」案件がなかったか確認（除外フィルターの過剰検出チェック）

## 5-4. 市場集計（半年〜3年の商品設計に直結）

- 相手が欲しかったものの分布（HP / LP / 導線 / 集客 / SNS / AI / 信頼形成）
- 「HPそのもの」より「集客・導線」を求めた比率
- → 比率の変化を追跡し、AI製作ツール普及でHP制作の価値が下がった場合に何を商品の中核に据えるかの判断材料とする

## 5-5. システム改訂判断

10件ごとに以下を必ず問う：

- OS①の除外フィルターに追加/削除すべき条件はあるか
- OS②の進行ラダー・赤信号定義に修正は必要か
- 事前仮説の的中率は上がっているか（上がっていれば事前判定の精密化を解禁してよい）
- このOS③自体に足りない検証項目はあるか
