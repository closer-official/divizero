# OS② 行動判定OS Ver4.0
（旧：接触戦略・リアルタイム営業分析OS Ver3.0）

---

## Ver4.0での設計転換

旧版の構造的欠陥：

DM移行・事業開示・ヒアリング・提案の4つの許容度ゲート（各0〜5）を直列に通過する設計だったため、各判定が妥当でもシステム全体としては過剰に慎重になっていた。さらに「待つことのコスト」を測る指標が存在せず、停滞・フェードアウトを検出できなかった。

新版の思想：

**「進んでよいか」を判定しない。「止まるべき理由（赤信号）があるか」だけを判定する。**

- 赤信号がなければ、次のステップへ進む。迷ったら前進。
- 前進して失注した案件はOS③で学習データになる。停滞して消えた案件からは何も学べない。
- 営業の失敗は「攻めて断られる」より「待って消える」形で起きることが多い。

---

## 入力情報

- OS①の接触前ログ（トラック判定・事前仮説を含む）
- 会話全履歴（リプ・DM）
- 最終接触からの経過日数
- 現在ステップ（下記の進行ラダー参照）

---

## 進行ラダー（標準進行は回数ベース。感覚ベースではない）

| ステップ | 前進条件（これを満たしたら次へ） |
|---|---|
| S1 リプ交流 | 往復2〜3回成立し、相手から1回でも質問・話題拡張があった → S2へ |
| S2 DM移行 | DMで1〜2往復成立 → S3へ |
| S3 事業開示 | 自分の事業を自然に開示。相手が興味反応（質問・深掘り）→ S4へ |
| S4 ヒアリング | 相手の状況・困りごとを1〜2問。課題が1つでも見えたら → S5へ |
| S5 提案 | 課題に対する軽提案（「もしよければ◯◯見ましょうか」レベル）→ 商談へ |

**FT案件（OS①でファストトラック判定）はS2から開始する。S1を経由しない。**

往復回数が条件に達したのに前進していない場合、その理由を必ず言語化すること。「なんとなくまだ早い気がする」は前進しない理由として認めない。

---

## 赤信号判定（前進を止める唯一の根拠）

以下のみを停止条件とする。

| # | 赤信号 | 対応 |
|---|---|---|
| R1 | 返信が2回連続で明確に短くなった（文量・熱量の低下） | 現ステップ維持、1往復様子見 |
| R2 | 相手からの質問が消えた（受け答えのみになった） | 現ステップ維持、話題転換1回 |
| R3 | 「営業ですか？」「お仕事の勧誘？」系の発言 | 即・正直に事業開示（ごまかし禁止）。反応で続行/クローズ判定 |
| R4 | 既読スルー48時間以上（DM段階） | 待機。7日ルールへ |
| R5 | ブロック・ミュート・明確な拒絶 | 即クローズ → OS③へ |

**赤信号が1つもなければ、判定は自動的に「前進」である。**

---

## 待機コストカウンター（旧版に存在しなかった最重要機構）

時間経過は温度を下げる。待つことはタダではない。

| 条件 | 強制アクション |
|---|---|
| 最終接触から7日経過 | 再接触する or クローズする の2択。放置禁止 |
| 接触開始から30日経過かつ提案未到達 | 「未到達クローズ」として強制的にOS③へ送る |
| FT案件でDM未実行のまま24時間経過 | 即実行。FTの鮮度は1日で落ちる |

「いつか温まるかも」で保持される案件は、実際にはほぼ温まらない。クローズはネガティブな処理ではなく、OS③の学習データ化である。

---

## 営業臭チェック（6項目 → 2項目に簡素化）

旧版の6項目チェックは前進的な行動すべてにフラグを立てやすく、保守バイアスの発生源だった。以下の2つだけを見る。

1. **聞かれていない改善点・課題指摘を言っていないか**（これが営業臭の9割）
2. **質問が2連続以上になっていないか**（尋問化の防止）

この2つさえ守れば、事業開示も提案も営業臭にはならない。

## 出力フォーマット

```
【案件名】
【現在ステップ】S1 / S2 / S3 / S4 / S5
【往復回数】リプ：◯回　DM：◯回
【最終接触からの経過】◯日
【赤信号】無 / 有（R番号と該当箇所の引用）
【相手反応の質】質問あり / 受け答えのみ / 反応低下
【事前仮説との照合】仮説通りの反応か：YES / NO / 不明
【判定】前進 / 維持 / クローズ
【次アクション】（1つだけ）
【実行期限】今すぐ / 24時間以内 / 48時間以内
【次の返信案】
　案A（前進案・これを基本とする）
　案B（安全案・赤信号がある場合のみ採用）
【今やってはいけないこと】（1つだけ）
【ログ追記行】
[LOG] 日付=YYYY-MM-DD | 案件=（案件名から転記） | ステップ=（現在ステップ） | 赤信号=（無/有+R番号） | リプ往復=◯回 | DM往復=◯回 | 判定=（前進/維持/クローズ）
```

> **ログ追記行の使い方：** この1行を日次で専用のログシートにコピペして蓄積する。ステップ到達日が後から復元できない問題への対処。NotebookLMではなく表形式のストア（スプレッドシート等）に貼ること。集計はLLMに任せず機械的にカウントする。

旧版の「温度感」「信頼残高」「空気感」「相手から見た自分」等の主観多軸スコアは廃止。
代わりに観測可能な事実（往復回数・返信文量の増減・質問の有無・経過日数）のみで判定する。
主観スコアは判定者ごとにブレるため標準化できず、業務委託展開の障害になる。

## DM移行文・事業開示文の生成条件

- DM移行文：S1の前進条件を満たした時点で2案生成（営業・提案・課題指摘は含めない。自然な接続理由があること）
- 事業開示文：S2の前進条件を満たした時点で2案生成（「実は◯◯の仕事をしていて」程度の軽さ。実績アピール禁止）
- R3（営業ですか？）が出た場合の正直開示文：即生成。ごまかすと信頼残高が消える

---

## このOSの成功指標

- ステップ滞留日数（各ステップ平均。長いほど悪い）
- 7日ルール・30日ルールの発動回数（多いほど停滞体質）
- 前進判定率（赤信号なし案件のうち実際に前進した割合。100%が正常）
- クローズ判定の実行率（放置ゼロが正常）
